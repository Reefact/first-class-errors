# FCE017: SensitiveDataInErrorContext

🌍 **Languages:**  
🇬🇧 English (this file) | 🇫🇷 [Français](./FCE017.fr.md)

| | |
|---|---|
| **Category** | Usage (`FirstClassErrors.Usage`) |
| **Severity** | 🟠 Warning |
| **Enabled by default** | No (opt-in) |

An `ErrorContextKey` is declared with a name that denotes a secret, credential, or piece of personal data (`password`, `token`, `secret`, `api key`, `connection string`, `credit card`, `SSN`, ...). Error context is copied into logs, dashboards, and the generated error catalog, so a secret placed there leaks wherever the error travels.

This is a **name heuristic**: it inspects the key name only. It cannot see the runtime value, so it neither proves a leak nor catches a secret hidden behind an innocuous key name — it flags declarations that *look* like they will carry sensitive data, so you can move them out of context.

## Noncompliant

```csharp
public static readonly ErrorContextKey<string> Password =
    ErrorContextKey.Create<string>("USER_PASSWORD"); // FCE017
```

```csharp
public static readonly ErrorContextKey<string> ApiKey =
    ErrorContextKey.Create<string>("API_KEY"); // FCE017
```

## Compliant

Keep the secret out of the context entirely, and attach only a non-sensitive reference that is safe to log:

```csharp
// A stable, non-sensitive identifier is enough to investigate.
public static readonly ErrorContextKey<string> CredentialId =
    ErrorContextKey.Create<string>("CREDENTIAL_ID");

// If a fragment must be shown, expose a redacted form, not the value itself.
public static readonly ErrorContextKey<string> CardLast4 =
    ErrorContextKey.Create<string>("CARD_LAST4");
```

See [Error Context — When not to use it](../ErrorContext.en.md#-when-not-to-use-it).

## Enabling this rule

This rule is disabled by default. Turn it on in `.editorconfig`:

```ini
dotnet_diagnostic.FCE017.severity = warning   # or suggestion / error
```

The term list is curated and cannot be exhaustive; treat a green build as "no obvious secret-named keys", not as a guarantee that context is secret-free. If the rule flags an intentional false positive (for example a key named after a *password policy* rather than a password), silence that line:

```csharp
#pragma warning disable FCE017 // this key carries a policy id, not a secret
public static readonly ErrorContextKey<string> PasswordPolicyId =
    ErrorContextKey.Create<string>("PASSWORD_POLICY_ID");
#pragma warning restore FCE017
```
