# FCE017: SensitiveDataInErrorContext

🌍 **Langues:**  
🇬🇧 [English](./FCE017.en.md) | 🇫🇷 Français (ce fichier)

| | |
|---|---|
| **Catégorie** | Usage (`FirstClassErrors.Usage`) |
| **Sévérité** | 🟠 Warning |
| **Activée par défaut** | Non (opt-in) |

Une `ErrorContextKey` est déclarée avec un nom qui désigne un secret, un identifiant d'authentification ou une donnée personnelle (`password`, `token`, `secret`, `api key`, `connection string`, `credit card`, `SSN`, …). Le contexte d'erreur est recopié dans les logs, les dashboards et le catalogue d'erreurs généré : un secret placé là fuite partout où l'erreur circule.

C'est une **heuristique sur le nom** : elle n'examine que le nom de la clé. Elle ne voit pas la valeur au runtime, donc elle ne prouve pas une fuite et ne détecte pas un secret caché derrière un nom anodin — elle signale les déclarations qui *semblent* devoir transporter une donnée sensible, pour que vous les sortiez du contexte.

## Non conforme

```csharp
public static readonly ErrorContextKey<string> Password =
    ErrorContextKey.Create<string>("USER_PASSWORD"); // FCE017
```

```csharp
public static readonly ErrorContextKey<string> ApiKey =
    ErrorContextKey.Create<string>("API_KEY"); // FCE017
```

## Conforme

Gardez le secret hors du contexte, et n'attachez qu'une référence non sensible, sûre à logger :

```csharp
// Un identifiant stable et non sensible suffit à l'investigation.
public static readonly ErrorContextKey<string> CredentialId =
    ErrorContextKey.Create<string>("CREDENTIAL_ID");

// Si un fragment doit être montré, exposez une forme masquée, pas la valeur.
public static readonly ErrorContextKey<string> CardLast4 =
    ErrorContextKey.Create<string>("CARD_LAST4");
```

Voir [Contexte d'erreur — Quand ne pas l'utiliser](../ErrorContext.fr.md#-quand-ne-pas-lutiliser).

## Activer cette règle

Cette règle est désactivée par défaut. Activez-la dans `.editorconfig` :

```ini
dotnet_diagnostic.FCE017.severity = warning   # ou suggestion / error
```

La liste de termes est choisie à la main et ne peut pas être exhaustive : un build vert signifie « aucune clé au nom manifestement secret », pas « le contexte est garanti sans secret ». Si la règle signale un faux positif intentionnel (par exemple une clé nommée d'après une *politique de mot de passe* et non un mot de passe), faites-la taire sur cette ligne :

```csharp
#pragma warning disable FCE017 // cette clé porte un id de politique, pas un secret
public static readonly ErrorContextKey<string> PasswordPolicyId =
    ErrorContextKey.Create<string>("PASSWORD_POLICY_ID");
#pragma warning restore FCE017
```
